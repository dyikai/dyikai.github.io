Swift Concurrency

一、Swift Concurrency与GCD(Grand Central Dispatch)的区别

Swift Concurrency 和 GCD(Grand Central Dispatch) 都是用于处理并发任务的技术，但它们在使用方式和设计理念上有一些明显的区别。

1. 设计理念

  • GCD: GCD 是基于线程池模型的低级别 API，允许开发者通过队列的方式来管理任务的执行。GCD 提供了一个全局的线程池，开发者可以将任务添加到不同的队列（如并发队列或串行队列）来异步执行。它更像是对底层线程和队列的直接控制。

  • Swift Concurrency: Swift Concurrency 是一种更高级的抽象，基于 async/await 机制，让开发者可以编写异步代码而无需显式地管理队列和线程。它通过 Actors 和 Task 模型来自动处理并发和同步问题，从而减少了并发错误的风险。

2. 编写风格

  • GCD： 使用 GCD 时，开发者需要处理大量的闭包和回调。代码会显得嵌套比较深，尤其是在复杂的异步逻辑中。这种风格容易导致 “回调地狱” 或者使代码的可读性变差。
  • Swift Concurrency： 使用 async/await，代码更加线性和直观，类似于同步代码的写法。它使得代码结构更清晰，并且大大减少了嵌套。

3. 错误处理

  • GCD： GCD 不直接支持错误处理，开发者必须自己在闭包中处理错误（如通过返回值或者自定义的回调方式）。
  • Swift Concurrency： Swift Concurrency 完全支持 Swift 的错误处理机制。使用 async/await 时，可以与 try 和 throws 结合使用，进行更自然的错误处理。

4. 任务取消

  • GCD： GCD 没有内置的任务取消机制。如果需要取消任务，开发者需要自己通过某种标志位或者逻辑处理来实现任务的手动取消。
  • Swift Concurrency： Swift Concurrency 支持任务的取消机制，可以通过 Task 来控制任务的取消。你可以使用 Task.checkCancellation() 检查任务是否被取消，并根据需要终止任务。

5. 数据同步

  • GCD： 在 GCD 中，开发者需要通过使用同步机制（如锁、信号量等）来确保共享资源的安全访问，容易出现死锁和竞争条件等问题。
  • Swift Concurrency： Swift Concurrency 引入了 actors 这种并发模型，来保证对状态的独占访问。通过 actor，开发者可以确保同一时间只有一个任务可以访问某个共享的状态。

6. 性能和调度

  • GCD： GCD 提供了队列的优先级设置，可以根据任务的优先级来调度线程池中的任务。开发者需要手动管理任务的执行顺序和线程的调度。
  • Swift Concurrency： Swift Concurrency 的调度器会根据系统资源和任务的优先级来智能地调度任务。开发者可以通过设置 TaskPriority 来控制任务的优先级，而不需要直接管理线程。

总结

  • GCD 更适合低级别的任务管理和调度控制，适合那些需要高度自定义和控制的场景。
  • Swift Concurrency 提供了一种更现代化、更安全的方式来处理并发任务，减少了并发编程中的常见问题，代码更加简洁易读，适合绝大多数需要并发执行的场景。

当我们调用一个异步方法时，执行在方法返回前会被中断。我们使用await关键字，将其写在调用前以标记可能出现的挂起点。

当我们使用await标记时，当前代码段可能在等待异步函数或方法返回是暂停执行，这也被称为放弃线程。因为在后台，Swift会暂停当前线程上的代码执行，并且在该线程上运行其他代码。因为带有await的代码需要能够暂停执行，所以只有程序中的某些地方可以调用异步函数或方法：

  • 异步函数、方法或属性体中的代码。
  • 标记为@main的结构、类或枚举的静态main()方法中的代码。
  • 非结构化子任务中的代码。

五、任务(Task)和任务组(Task Group)

任务是可以作为程序的一部分异步运行的工作单元。所有异步代码都作为某个任务的一部分运行。任务本身一次只做一件事，但是当你创建多个任务时，Swift可以安排它们同时运行。

上一节中使用的async-let语法隐式地创建了一个子任务——当我们已经知道程序需要运行哪些任务时，这种语法会很好地工作。我们还可以创建一个任务组(TaskGroup的一个实例)并显式地向该组添加子任务，这样可以更好地控制优先级和取消，并且可以创建不定量的任务。

任务按层次结构排列。给定任务组中的每个任务都有相同的父任务，每个任务可以有子任务。由于任务和任务组之间存在显式关系，因此这种方法称为结构化并发。任务之间显式的父子关系有几个优点:

  • 在父任务中，不能忘记等待它的子任务完成。
  • 当在子任务上设置更高的优先级时，父任务的优先级将自动升级。
  • 当父任务被取消时，它的每个子任务也会自动取消。
  • 本地任务的值有效且自动地传播到子任务。

任务

任务允许我们从非并发方法创建并发环境，使用 async/await 调用方法。

我们可以通过Task(priority: TaskPriority?, operation: () async -> Sendable)来创建一个任务，其中 TaskPriority 参数用于指定任务的优先级。优先级决定了系统在调度和执行任务时如何分配资源，特别是在多任务执行时，优先级较高的任务可能会优先获得执行机会。

TaskPriority 是一个枚举类型，定义了任务的优先级。它包含以下几种常见的优先级选项：

  • .high：高优先级任务，通常用于需要立即响应的任务。例如，用户界面的更新任务。
  • .medium：中等优先级任务，适用于一般任务。默认情况下，许多任务会使用此优先级。
  • .low：低优先级任务，适用于可以延迟执行的任务。例如，后台数据同步或预加载任务。
  • .background：后台任务优先级，用于那些不需要用户立即注意的任务。适用于后台处理或预加载数据。
  • .userInitiated：用户发起的任务优先级，通常用于用户直接触发的任务，比如用户按下按钮后启动的操作。优先级略高于 .medium。
  • .utility：实用任务优先级，适用于需要一定时间完成，但不需要立即响应的任务。比如下载数据或处理文件。

任务的取消

Swift并发使用协作取消模型。每个任务在执行过程中检查自己是否在适当的时间点被取消，并对取消做出适当的响应。根据任务正在执行的工作，响应取消通常意味着以下其中一种:

  • 抛出一个类似CancellationError的错误
  • 返回nil或空集合
  • 返回部分完成的工作

如果图片很大或网络很慢，下载图片可能需要很长时间。为了让用户停止此工作，而无需等待所有任务完成，需要检查任务是否被取消，如果被取消则停止运行。任务有两种方法可以做到这一点:通过调用Task.checkCancellation()类型方法，或者通过读取Task.isCancelled类型属性。如果任务被取消，调用checkCancellation()将抛出错误;抛出任务可以将错误传播到任务之外，从而停止任务的所有工作。这样做的优点是易于实现和理解。为了获得更大的灵活性，可以使用isCancelled属性，它允许您执行清理工作，作为停止任务的一部分，如关闭网络连接和删除临时文件。

六、Actors

1. 初识Actor

Actor 是引用类型，简而言之，这意味着副本引用的是同一块数据。因此，修改副本也会修改原始实例，因为它们指向同一个共享实例。

然而，与类相比，Actor有两个重要的区别：

1. 它们不支持继承

2. 由 Actor 的主要职责决定，它们可以隔离对数据的访问

Actor一次只允许一个任务访问其可变状态，这使得多个任务中的代码可以安全地与参与者的相同实例进行交互。例如，这里有一个记录温度的actor:

当我们创建了一个实例并访问其属性时，这里产生了一些区别：

  • 我们可以直接访问实例的常量，因为在读取的过程中它的值不会被改变，是线程安全的。
  • 对于变量来说，我们的其他方法或属性会改变一个引用类型的可变状态，为了防止产生数据竞争，在actor作用域外需要同步访问，因此在对变量的操作需要添加await关键字。然而，对于actor作用于内的访问，由于actor本身是现成安全的，我们无需添加此关键字。

2. Actor 内的非隔离(nonisolated)访问

当我们知道actor的某一个计算属性或方法并未访问任何隔离数据时，swift为我们提供了nonisolated关键字来进行标识，这时我们就可以直接调用它们而无需将程序挂起。

有一个重要的点，对于常量来说，我们无需对它们进行标记，因为在它们本身就是安全的，当我们用nonisolated对它们进行标记时，编译器会报错来提醒我们。

3. 为什么在使用 Actors 时仍会出现数据竞争

在 Swift 中，actor 是为了解决并发编程中的数据竞争问题而引入的，它通过串行化对其内部状态的访问，确保不同的任务在并发环境中不会同时修改或读取相同的资源。然而，即使使用 actor，在某些情况下仍然可能会出现数据竞争，通常是由于以下原因：

a. actor 外部共享的可变数据

即使数据在 actor 内部受保护，如果某个属性是对外部共享的可变数据（如 class 或引用类型的数据），actor 也无法完全控制这些数据在外部的修改，可能会导致数据竞争。

在这个例子中，shared 是 SharedData 类型的对象。虽然 actor 可以保护对 shared 的引用，但它无法保护 SharedData 内部的数据。因为 SharedData 是引用类型，在其他任务可以并发访问 shared.value 时，可能会出现数据竞争。

b. 多个 actor 共享的状态

如果多个 actor 共享相同的状态（例如通过引用类型共享数据），就会产生数据竞争。actor 只能保护它内部的独立状态，而无法控制其他 actor 对共享资源的修改。

在这个例子中，ActorA 和 ActorB 都共享了同一个 SharedState 实例。虽然两个 actor 本身是隔离的，但它们操作的是同一块内存，因此并发修改 count 时仍然可能出现数据竞争。

c. 异步任务在 actor 内部并发执行

actor 默认通过串行化对其内部状态的访问来保证数据安全，但是如果你在 actor 内部使用了 Task 或其他并发操作，这些操作本身仍可能并发运行，导致数据竞争。

在这个例子中，虽然 Counter 是一个 actor，但内部创建了两个并发的任务，它们可能同时访问 count，导致数据竞争。由于这两个任务并发执行，actor 本身无法保证这两个任务的顺序，因此可能出现数据竞争。

4.如何避免数据竞争？

尽管 actor 能有效防止大部分数据竞争，但仍需要开发者遵守一些规则来避免潜在问题：$\$

  1. 避免在 nonisolated 方法中访问可变状态：尽量避免在 nonisolated 方法或属性中访问可能被并发修改的状态。如果确实需要访问，最好将这些方法设置为 isolated，或确保它们只访问不可变的状态。$\$
  2. 避免共享引用类型的可变数据：尽量避免在多个 actor 或异步任务之间共享同一个可变的引用类型对象。使用值类型（如 struct）或者通过 actor 的内部逻辑来管理对引用类型的访问。$\$
  3. 谨慎使用 Task 创建并发操作：在 actor 内部启动异步任务时，要注意这些任务可能会并发运行。通常，应该让 actor 自己管理任务的顺序，避免多个并发任务同时操作共享状态。$\$
  4. 使用其他同步机制：在某些需要额外控制并发访问的场景下，可能需要引入其他同步机制（如锁、信号量）来进一步保护共享资源的并发访问。
